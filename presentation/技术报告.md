# 基于 TCP 通信的联邦学习系统技术报告

## 1. 项目概述

本项目实现了一个面向 MNIST 分类任务的联邦学习实验框架。系统以 Python 原生 `socket` 为通信基础，使用 PyTorch 进行模型训练，核心目标是验证在 TCP 字节流通信条件下，多个边缘 Client 如何与中心 Server 协同训练一个全局模型，并在传输大容量模型参数时正确处理粘包、半包和大 payload 接收问题。

项目基础模式为集中式联邦学习：1 个 Server 与 3 个 Client 通过本地环回地址 `127.0.0.1` 建立 TCP 连接。Server 维护全局 CNN 模型，Client 各自加载本地拆分数据，完成本地训练后上传权重，Server 进行加权平均聚合。项目还实现了 Ring 去中心化拓扑、SplitFed 分割联邦、弱网延迟与掉队者容错、实时监控面板和可选 gzip 压缩等扩展。

## 4. 数据与模型

### 4.1 数据隔离

数据拆分后存放在 `data/splits/`：

| 文件 | 使用方 |
| --- | --- |
| `client_1_data.pt` | `client_1` 或 Ring Node 1 |
| `client_2_data.pt` | `client_2` 或 Ring Node 2 |
| `client_3_data.pt` | `client_3` 或 Ring Node 3 |
| `server_test_data.pt` | Server 或 Ring Node 1 的全局评估 |

Client 的 `_load_local_data` 根据自己的 `client_id` 推导文件名，例如 `client_2` 只能加载 `client_2_data.pt`。这满足作业中“Client 节点严禁越权读取全局数据或其他节点数据”的要求。Server 不加载训练数据，只加载 `server_test_data.pt` 进行全局测试。

### 4.2 CNN 模型

基础模型定义在 `model.py`，结构为：

1. `Conv2d(1, 32, kernel_size=3, padding=1)` + ReLU + MaxPool。
2. `Conv2d(32, 64, kernel_size=3, padding=1)` + ReLU + MaxPool。
3. 展平为 `64 * 7 * 7`。
4. `Linear(64 * 7 * 7, 128)` + ReLU。
5. `Linear(128, 10)` 输出 MNIST 10 类 logits。

SplitFed 模式将该 CNN 拆成两段：Client 侧只保留第一层卷积和池化，Server 侧保留第二层卷积、全连接层和分类头。

## 5. TCP 通信协议与粘包处理

### 5.1 封包结构

所有模式统一使用 `core/communicator.py` 中的 `TCPCommunicator`。封包格式如下：

| 字段 | 长度 | 内容 |
| --- | ---: | --- |
| Length Header | 8 字节 | 大端无符号整数，表示 payload 字节数 |
| Magic Number | 4 字节 | 固定为 `SF26` |
| Payload | L 字节 | `pickle.dumps(message_dict)` 的结果，可选 gzip 压缩 |

该设计把 TCP 字节流重新切分为应用层消息。Length Header 用于确定本次消息体的准确长度，Magic Number 用于发现错位或损坏的数据包，Payload 则承载注册、模型权重、SplitFed 激活值、梯度、shutdown 等不同类型消息。

### 5.2 发送流程

发送端调用 `send_data(sock, message_dict)`：

1. 使用 `pickle.dumps` 将 Python 字典序列化为字节流。
2. 若 `network.compression = true`，再使用 `gzip.compress` 压缩。
3. 用 `struct.pack('>Q', len(serialized_data))` 生成 8 字节长度头。
4. 使用 `sock.sendall(length + magic + payload)` 一次性发送完整应用层消息。

`sendall` 保证尽可能发送完整缓冲区，但 TCP 仍可能在网络层拆分为多段。因此接收端不能假设一次 `recv` 就得到完整消息。

### 5.3 接收流程

接收端调用 `recv_data_with_meta(sock)`：

1. `_recvall(sock, 8)` 精确读取长度头。
2. `struct.unpack('>Q', raw_msglen)[0]` 得到 payload 长度 L。
3. `_recvall(sock, 4)` 精确读取魔数并校验是否为 `SF26`。
4. `_recvall(sock, L)` 通过 while 循环累计接收，直到正好收满 L 字节。
5. 若启用压缩则解压，最后用 `pickle.loads` 反序列化为消息字典。

这个流程直接解决了 TCP 粘包和半包问题：即使模型参数被拆成很多 TCP 分片，只要连接不断开，接收端最终都会拼出完整 payload；即使多个消息连续到达，也会按长度头逐个解析，不会串包。

## 6. 集中式联邦学习模块

### 6.1 连接注册

Server 启动后创建 TCP 监听 socket，绑定 `topology.server.host:port`，并在 `_accept_loop` 中不断接受 Client 连接。每个连接交给一个独立线程 `handle_client` 处理。

Client 启动后连接 Server，并发送：

```python
{"type": "register", "client_id": "client_1"}
```

Server 收到后将 `client_id -> conn` 保存到 `active_clients`，回复 `register_ack`，并上报 `topology_update`。Server 在 `_wait_for_all_clients` 中阻塞等待，直到注册数量达到配置中的 Client 数量。

### 6.2 全局模型广播

每轮开始时，Server 调用 `_run_round_centralized(round_id)`。它构造 `round_start_centralized` 消息：

```python
{
  "type": "round_start_centralized",
  "round": round_id,
  "weights": global_model.state_dict(),
  "local_epochs": local_epochs,
  "lr": client_lr,
  "momentum": momentum,
  "weight_decay": weight_decay
}
```

随后 `_broadcast` 遍历当前 `active_clients`，将该消息逐个发送给 Client。每次发送都会记录 payload 大小、消息类型、累计发送字节数和消息数。

### 6.3 Client 本地训练

Client 收到 `round_start_centralized` 后：

1. 用 Server 下发的 `weights` 覆盖本地模型参数。
2. 根据消息中的学习率、动量、权重衰减创建 SGD 优化器。
3. 在自己的 `train_loader` 上训练配置的 `local_epochs`。
4. 每个 batch 上报 `batch_progress`，包括 batch loss、batch accuracy、当前轮次等。
5. 训练结束后分别计算本地 train、val、test 指标。
6. 返回 `model_update`，包含训练后的 `state_dict`、本地样本数和指标。

返回消息结构中最关键的是：

```python
{
  "type": "model_update",
  "round": round_id,
  "client_id": client_id,
  "weights": trained_state_dict,
  "num_samples": local_train_sample_count
}
```

`num_samples` 用于 Server 做加权平均，而不是简单平均。

### 6.4 Server 更新收集与 FedAvg 聚合

Server 的每个 Client 线程收到 `model_update` 后，将其写入 `round_updates[round_id][client_id]`，然后通知条件变量 `update_cv`。主训练线程通过 `_wait_round_updates(round_id)` 等待本轮 Client 更新。

聚合函数 `_aggregate_weighted(updates)` 按样本数进行 FedAvg：

```text
global_weight[k] = sum_i (num_samples_i / total_samples) * client_weight_i[k]
```

完成聚合后，Server 用聚合权重覆盖 `global_model`，再在 `server_test_data.pt` 上计算全局 `test_loss` 和 `test_acc`。如果 `test_acc >= target_accuracy`，Server 上报 `target_reached` 并向所有 Client 广播 shutdown。

## 7. 弱网与掉队者处理

弱网处理是本项目需要重点说明的模块。它分为 Client 侧模拟和 Server 侧容错两部分。

### 7.1 弱网配置入口

配置位于 `config.json` 或 `config/centralized.json` 的 `network.stragglers`：

```json
"stragglers": {
  "client_1": {"delay": 0.0, "drop_rate": 0.0},
  "client_2": {"delay": 5.0, "drop_rate": 0.0},
  "client_3": {"delay": 0.0, "drop_rate": 0.0}
}
```

`delay` 表示该 Client 在发送每轮更新前人为等待多少秒；`drop_rate` 表示该 Client 在某轮直接不上传更新的概率。默认配置中 `client_2` 每轮延迟 5 秒，模拟弱网或慢设备。

### 7.2 Client 侧延迟与丢弃

Client 初始化时读取自己的弱网配置：

```python
self.straggler_config = config["network"]["stragglers"].get(client_id, ...)
```

每轮训练前先调用 `_should_drop_round()`。如果随机数小于 `drop_rate`，该 Client 本轮直接跳过训练结果上传，并上报 `round_dropped`。如果不丢弃，则正常训练，训练完成后调用 `_maybe_delay()`：

```python
delay = straggler_config["delay"]
if delay > 0:
    monitor.post("straggler_delay", delay_seconds=delay)
    time.sleep(delay)
```

因此弱网延迟发生在“本地训练完成之后、模型更新发送之前”。这能真实模拟同步式联邦学习中常见的 straggler 问题：其他 Client 已经训练完并上传，但 Server 仍需等待慢节点返回。

### 7.3 Server 侧超时等待

Server 不直接在 `recv` 线程中做聚合，而是将各 Client 的更新存入 `round_updates`。主线程调用 `_wait_round_updates(round_id)` 等待本轮结果。该函数的核心逻辑是：

1. 记录开始时间和 deadline：`deadline = now + server_timeout`。
2. 在条件变量上等待，直到收齐所有 Client 更新或到达 deadline。
3. 超时后统计 `expected - responded`，即本轮没有返回的 Client。
4. 对未返回 Client 上报 `straggler_dropped`。
5. 判断 `len(got) >= min_clients`，决定本轮是否仍能继续。

如果收到更新数不少于 `min_clients`，Server 会只用已收到的 Client 更新做 FedAvg；如果收到数低于 `min_clients`，Server 上报 `round_aborted`，设置 `stop_reason = insufficient_client_updates` 并终止训练。

### 7.4 当前配置下的实际效果

默认 `config.json` 中 `client_2.delay = 5.0`，但 `server_timeout = 300.0`；`config/centralized.json` 中 `server_timeout = 15.0`。两者都大于 5 秒，所以正常实验中 Server 会记录慢节点延迟，但通常不会丢弃 `client_2`。测试报告中也显示 Centralized 和 SplitFed 都记录了 `client_2` 延迟，但 `dropped_clients = 0`。

如果现场想展示丢弃效果，可以将 `server_timeout` 调小到小于 5 秒，例如 3 秒，或者将 `client_2.drop_rate` 设置为 1.0。此时 Server 会在该轮上报 `straggler_dropped`，并在收到至少 `min_clients` 个更新时继续聚合。

## 8. 监控与日志模块

系统监控分为事件上报、API 聚合、可视化三层。

### 8.1 事件上报

`MonitorReporter` 使用标准库 `urllib.request` 向 `http://host:port/report` POST JSON 事件。事件中包含：

| 字段 | 含义 |
| --- | --- |
| `ts` | 时间戳 |
| `source` | 上报来源，如 `server`、`client_1`、`ring_node_2` |
| `event_type` | 事件类型，如 `network_io`、`round_end` |
| `seq` | 来源内递增序号 |
| 其他字段 | 轮次、payload 大小、准确率、耗时、客户端 ID 等 |

### 8.2 网络日志

Server、Client、RingNode 每次 `_send` 或 `_recv` 都记录网络事件：

1. 方向：`out` 或 `in`。
2. peer：对端 Client 或 Server。
3. message_type：如 `round_start_centralized`、`model_update`、`split_batch`。
4. payload_label：人类可读标签，如 `model_weights_update`。
5. payload_bytes：本次 payload 字节数。
6. 累计发送/接收字节数与消息数。
7. round：所属训练轮次。

这满足作业中“控制台实时输出轮次、Client_ID、广播耗时、接收耗时、Payload 大小等网络日志”的要求。项目还把这些事件保存到 `logs/<run_id>/monitor-events.jsonl`，并生成 `monitor-summary.json` 与训练曲线。

### 8.3 可视化

`utils/monitor_api.py` 提供 FastAPI 服务，包括健康检查、日志查询、summary 查询、训练启停等接口。前端位于 `web/`，可展示拓扑、事件日志、网络吞吐、loss/accuracy 曲线和慢节点统计。远程更新后的 `Image/` 目录已经包含三种模式的监控截图和曲线图，可直接用于 poster。

## 9. Ring 去中心化扩展

Ring 模式对应作业扩展“去中心化通信”。该模式废弃中心 Server，三个节点组成环：

```text
Node 1 -> Node 2 -> Node 3 -> Node 1
```

每个 RingNode 同时具备监听和发送能力：

1. 启动时绑定自己的端口，等待前驱节点连接。
2. 计算自己的后继节点地址。
3. Node 1 作为 initiator，先训练本地模型并传给 Node 2。
4. Node 2 收到模型后加载权重，本地训练，再传给 Node 3。
5. Node 3 同样训练后传回 Node 1。
6. Node 1 收到绕环一周后的模型，在 `server_test_data.pt` 上做全局评估。

Ring 模式中没有 FedAvg 的中心聚合，而是模型在各节点间顺序更新。它更接近循环式参数传递，优点是不依赖中心 Server，缺点是节点之间存在严格串行依赖：一个节点慢，会影响整个环的推进。

弱网在 Ring 中同样通过 `network.stragglers.client_x` 生效。`node_2` 对应 `client_2`，因此默认也会延迟 5 秒。区别在于 Ring 中没有中心 Server 的 `min_clients` 聚合容错，节点主要通过接收超时、发送重试和 shutdown 通知来保障流程可结束。

## 10. SplitFed 扩展

SplitFed 模式把模型拆分成 Client 侧和 Server 侧。训练时 Client 不上传完整模型参与普通前向传播，而是：

1. Client 用 `SplitClientCNN` 计算中间激活值 `smashed`。
2. Client 发送 `split_batch`，其中包含激活值和标签。
3. Server 用 `SplitServerCNN` 完成后半段前向传播，计算 loss。
4. Server 对激活值求梯度，并发送 `split_grad` 返回 Client。
5. Client 用返回梯度继续反向传播，更新 Client 侧模型。
6. 每轮结束后，Client 上传前半段模型权重，Server 加权聚合 Client 侧模型。

SplitFed 的优点是 Client 不需要持有完整模型结构，且 Server 可以参与每个 batch 的训练计算。缺点是通信频率从“每轮传一次模型权重”变成“每个 batch 传激活值和梯度”，所以网络开销大幅增加。测试报告中 SplitFed 约 9.0 GB 收发量，明显高于 Centralized 的约 58.0 MB。

## 11. 实验结果

远程更新后的 `测试报告.md` 记录了三种模式的主要结果：

| 模式 | 完成轮次 | 最终测试准确率 | 最终测试损失 | 运行时长 | 网络发送量 | 网络接收量 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| Centralized | 6 / 10 | 0.9900 | 0.0295 | 192.5s | 58.0 MB | 58.0 MB |
| Ring | 9 / 10 | 0.9901 | 0.0341 | 214.1s | 43.5 MB | 43.5 MB |
| SplitFed | 4 / 10 | 0.9904 | 0.0297 | 334.2s | 9.0 GB | 9.0 GB |

三种模式均达到约 99% 的 MNIST 测试准确率，显著超过作业要求的 85%。Centralized 模式总耗时最短，适合作为基础功能展示；Ring 模式证明了去中心化 TCP 拓扑可行；SplitFed 收敛轮次最少，但通信量最高，能很好体现网络架构对通信效率的影响。
